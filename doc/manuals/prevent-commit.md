# Налаштування локального блокування комітів із секретами (Gitleaks Pre-commit Hook)

Цей мануал описує процес встановлення та налаштування локального Git-хука (`pre-commit`), який автоматично сканує змінені (staged) файли на наявність чутливих даних (API-ключів, паролів, токенів тощо) перед кожним комітом.

Якщо хук виявляє секрет, створення коміту блокується. Це дозволяє попередити потрапляння конфіденційної інформації до історії Git ще до відправки (push) до віддаленого репозиторію.

---

## 🛠️ Як це працює

1. Коли ви запускаєте команду `git commit`, Git автоматично виконує скрипт `.git/hooks/pre-commit`.
2. Скрипт перевіряє значення конфігурації `hooks.gitleaks`. Якщо воно встановлено в `true`, перевірка запускається.
3. Скрипт шукає встановлену утиліту `gitleaks` у директорії `~/.local/bin/gitleaks` (або `gitleaks.exe` у Windows).
4. **Автоматичне встановлення:** Якщо утиліту не знайдено, скрипт самостійно завантажує найсвіжішу версію Gitleaks із офіційних релізів GitHub відповідно до вашої операційної системи та архітектури процесора (macOS/Linux/Windows).
5. Скрипт виконує команду `gitleaks protect -v --staged --redact`, перевіряючи лише ті файли, що підготовлені до коміту (`git add`).
6. Якщо знайдено секрети, утиліта повертає код помилки `1`, скрипт виводить попередження `COMMIT REJECTED` і скасовує створення коміту.

---

## 🚀 Налаштування (Крок за кроком)

### Крок 1: Копіювання скрипту хука
Усі Git-хуки локального репозиторію зберігаються у директорії `.git/hooks/`, яка не відстежується Git. Щоб налаштувати хук, скопіюйте шаблон скрипту з версіонованого каталогу репозиторію:

```bash
cp scripts/pre-commit .git/hooks/pre-commit
```

### Крок 2: Надання прав на виконання
Скрипт повинен мати права на виконання, щоб Git міг його запустити:

```bash
chmod +x .git/hooks/pre-commit
```

### Крок 3: Увімкнення перевірки в Git
За замовчуванням хук вимкнений, щоб не заважати розробникам до початку налаштування. Увімкніть його за допомогою локальної конфігурації Git:

```bash
git config hooks.gitleaks true
```

*Примітка: Цей параметр записується у файл `.git/config` вашого локального репозиторію.*

---

## 🔍 Перевірка працездатності

1. Спробуйте створити новий коміт (наприклад, додавши звичайні зміни). Під час виконання ви побачите повідомлення:
   ```text
   pre-commit hook running
   Gitleaks installed version: 8.30.1
   ```
2. Щоб протестувати блокування, додайте тимчасовий тестовий секрет (наприклад, рядок з паролем `password: "supersecretpass"`) у будь-який YAML-файл, додайте його (`git add`) та спробуйте закомітити.
3. Ви повинні отримати наступний вивід, а коміт буде відхилено:
   ```text
   pre-commit hook running
   Gitleaks installed version: 8.30.1
   ...
   Warning:  COMMIT REJECTED  Gitleaks has detected sensitive information in your changes.
   
               To disable the gitleaks precommit hook run the following command:
   
               git config hooks.gitleaks false
   ```

## ⚙️ Управління конфігурацією

* **Тимчасове вимкнення хука:**
  Якщо вам необхідно тимчасово вимкнути перевірку (наприклад, для тестових комітів у локальній пісочниці):
  ```bash
  git config hooks.gitleaks false
  ```
  Для повторного ввімкнення виконайте:
  ```bash
  git config hooks.gitleaks true
  ```

* **Разовий обхід перевірки (Bypass):**
  Ви можете змусити Git проігнорувати хуки для конкретного коміту, використавши прапорець `--no-verify`:
  ```bash
  git commit -m "my commit message" --no-verify
  ```
  *(Використовуйте цей варіант з обережністю, оскільки він підвищує ризик витоку секретів у віддалену гілку).*

## 🌐 Налаштування блокування на рівні GitHub (Remote Protection)

Для забезпечення максимальної безпеки (підхід "Defense in Depth") локальних перевірок недостатньо, оскільки розробник може випадково або навмисно обійти хуки. Необхідно налаштувати блокування витоку секретів безпосередньо в репозиторії на **github.com**.

### 1. Увімкнення GitHub Push Protection
GitHub має вбудований безкоштовний функціонал сканування на наявність секретів, який може блокувати спроби пушу (git push), якщо коміти містять відомі секрети (наприклад, ключі Google Gemini, OpenAI, AWS тощо).

**Як увімкнути:**
1. Перейдіть у ваш репозиторій на **github.com**.
2. Відкрийте вкладку **Settings** (Налаштування) вгорі.
3. У лівому боковому меню виберіть **Security** -> **Code security** (або **Code security and analysis**).
4. Знайдіть розділ **Secret scanning** (Сканування секретів) та увімкніть його.
5. Під ним увімкніть опцію **Push protection** (Захист при push).

*Після цього при спробі виконати `git push` із секретом GitHub відхилить команду push та надішле розробнику інструкцію, як вилучити секрет з історії.*

### 2. Налаштування правил захисту гілок (Branch Protection Rules)
Щоб гарантувати, що вразливі комміти або зміни, які не пройшли перевірку якості та безпеки, не потраплять у гілки `dev` та `main`, налаштуйте обов'язкове проходження перевірок (Status Checks) перед злиттям Pull Request:

**Як налаштувати:**
1. У вкладці **Settings** репозиторію виберіть ліворуч **Code and automation** -> **Branches** (Гілки).
2. Натисніть **Add branch protection rule** (або відредагуйте існуюче для гілок `main` та `dev`).
3. Задайте шаблон імені гілки (наприклад, `main` або `dev`).
4. Увімкніть опцію **Require status checks to pass before merging** (Вимагати успішного проходження перевірок стану перед злиттям).
5. У пошуковому рядку перевірок стану знайдіть та додайте назву джоби з нашого робочого процесу GitHub Actions:
   - `Gitleaks, Lint, Typecheck, Evals` (це назва джоби `quality-gate` у нашому файлі [.github/workflows/cicd.yml](file:///Users/pokhrime/work/Docs/Tranings/DevOpsIntensive/Hackathon/hackathon-devops/.github/workflows/cicd.yml)).
6. Увімкніть **Require branches to be up to date before merging** (якщо потрібно).
7. Натисніть **Save changes** (Зберегти зміни).

*Це заблокує кнопку Merge в Pull Request, якщо Gitleaks чи Evals Quality Gate завершилися помилкою в CI/CD пайплайні.*

> [!IMPORTANT]
> **Автоматичні коміти у CI/CD (Bypass Branch Protection):**
> Оскільки наш CI/CD пайплайн виконує автоматичні коміти в гілку `dev` (оновлення тегів образів та синхронізація промптів), дефолтний токен `GITHUB_TOKEN` не зможе зробити `git push` через правила захисту гілок.
> 
> Для обходу захисту гілок у CI/CD використовується секрет `ADMIN_PAT`. Необхідно налаштувати його наступним чином:
> 
> **1. Створення Fine-grained Personal Access Token (PAT):**
> * Перейдіть у **Settings** вашого профілю GitHub -> **Developer settings** -> **Personal access tokens** -> **Fine-grained tokens**.
> * Натисніть **Generate new token**.
> * У полі **Repository access** виберіть **Only select repositories** та вкажіть ваш репозиторій `hackathon-devops`.
> * У секції **Permissions** -> **Repository permissions** встановіть такі доступи:
>   * **Contents**: виберіть **Access: Read and write** (для запису/комітів).
>   * **Metadata**: виберіть **Access: Read-only** (обов'язкове системне поле).
> * Згенеруйте та скопіюйте отриманий токен.
> 
> **2. Додавання секрету в репозиторій:**
> * Перейдіть у налаштування репозиторію на GitHub: **Settings** -> **Secrets and variables** -> **Actions**.
> * Натисніть **New repository secret**.
> * Назва секрету: `ADMIN_PAT`.
> * Значення секрету: вставте ваш скопійований токен.
> * Натисніть **Add secret**.
