## 2. Хронологічна структура імплементації інфраструктури

### Крок 1: SRE-налаштування та безпека хоста (Фаза 1)

-   **Мережева ізоляція VM:** Для захисту інфраструктури хакатону на віртуальній машині активовано локальний фаєрвол `ufw`.
    
-   **Закриття портів:** Блокування зовнішніх портів, окрім легітимного доступу для розробників за SSH-ключами. Host-порти `80`, `443` та `8081` зайняті системними сервісами edge-вузла.
    

### Крок 2: Топологія та розділення контурів (Фаза 1–2)

-   Локальний Kubernetes-рушій K3d містить два ізольовані кластери: `jobmatch-stage` та `jobmatch-prod`.
    
-   Нові інфраструктурні модулі безпеки та маршрутизації виділені **виключно в стейдж-папку (`clusters/dev/`)**, залишаючи продакшен повністю чистим.
    

### Крок 3: Синхронізація Git-історії, образів та розв'язання кризи авторизації

-   Виконано pull-ребейз (`git pull upstream dev --rebase`) для об'єднання інфраструктурних файлів із коммітами бекенду від `mkdir28`.
    
-   **Виправлення авторизації FluxCD:** Виявлено конфлікт прав доступу (Collaborator) у репозиторії Марини (`mkdir28`), що блокував роботу за SSH-ключами. Базовий секрет `flux-system` успішно переналаштовано на ручний HTTPS-формат із використанням Personal Access Token розробника `kylib4444`. **Flux успішно синхронізувався з оригінальним репозиторієм на коміті `c785630`**.
    

### Крок 4: Інтеграція з Agentic Sandbox (abox) та Prompt-безпека (Фаза 2)

-   У кластер було вручну імпортовано Custom Resource Definition (CRD) для типу `promptenrichments.gateway.abox.ai`.
    
-   Створено об'єкт `system-prompt-protection` в просторі назв `jobmatch-dev`, який інжектує системні інструкції та захищає логіку Шлюзу Агентів.
    

### Крок 5: Розчеплення ступору та стабілізація `external-secrets`

-   Під час спроби автоматичного оновлення до версії **`0.20.4`** Helm-контролер упав у термінальну помилку `UpgradeFailed` через конфлікт валідації старих CRD (`spec.conversion.strategy: Required value`).
    
-   **Рішення:** Старі невалідні CRD були примусово видалені з API-сервера. Базу даних застряглого релізу (Helm Secrets) повністю випалено з простору назв `flux-system` за допомогою `kubectl delete secrets -n flux-system -l owner=helm`.
    
-   Після очищення пам'яті Flux запустив чисту інсталяцію (`install action`) з нуля. **Компонент успішно оновлено до версії `0.20.4` (Статус `READY: True`)**.
    

### Крок 6: Стабілізація мережі через Traefik Ingress та налаштування доступу розробника

-   Розгорнуто маніфест `agent-ingress`, який отримав IP-адресу балансувальника `172.18.0.2`.
    
-   **Рішення для розробників:** Оскільки адреса Ingress є внутрішньою сірою IP-адресою в Docker-мережі кластера (`172.18.0.2`), а порти `80/443` на хості зайняті, розроблено залізобетонну схему доступу до вебморди в обхід обмежень Traefik. Доступ реалізовано через комбінацію локального Kubernetes проксі на вільний порт хоста `8080` та SSH-тунелювання.
    

## 3. Файлова архітектура репозиторію та локалізація конфігів

Маніфести контурів GitOps суворо структуровані відповідно до стандартів декларативного керування FluxCD:

**Шлях до файлу/директорії в platform/flux/clusters/**

**Цільове призначення та опис конфігурації**

**`dev/`**

**Контур тестування (Stage-кластер `k3d-jobmatch-stage`)**.

├── `apps/jobmatch/helm-release.yaml`

Специфікація та параметри релізу додатків Scout (API та Web).

├── `infrastructure/agent-gateway/`

Новий модуль маршрутизації, шлюзування та AI-безпеки.

│ ├── `kustomization.yaml`

Складальний файл Kustomize для об'єднання ресурсів шлюзу.

│ ├── `ingress.yaml`

Маніфест Traefik Ingress для ізольованого роутингу точки входу `/api`.

│ └── `prompt-filters.yaml`

Конфігурація об'єкта `PromptEnrichment` для фільтрації промптів через abox.

└── `infra.yaml`

Кореневий бутстрап Flux для Stage (декларує завантаження папок контуру).

**`prod/`**

**Чистий стабільний контур (Production-кластер)**.

├── `apps/jobmatch/helm-release.yaml`

Конфігурація релізу додатків для Prod-середовища.

└── `infra.yaml`

Кореневий бутстрап Flux для Prod (ізольований від експериментальних модулів).

## 4. Специфікація конфігураційних маніфестів (Джерела правди)

### Модуль Маршрутизації: `ingress.yaml`

YAML

```
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: agent-ingress
  namespace: jobmatch-dev
  annotations:
    traefik.ingress.kubernetes.io/router.entrypoints: web
spec:
  rules:
  - http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: jobmatch-dev-api
            port:
              number: 8080

```

### Схема Локального Підключення Розробника (Developer Access Matrix)

Для безпечного підключення до вебморди `localhost:8080` з локальної робочої станції використовується наступний двокроковий ланцюжок порту та тунелю:

1.  **Крок 1 (Запуск проксі-труби на Stage-сервері):**
    
    Bash
    
    ```
    kubectl port-forward svc/jobmatch-dev-web -n jobmatch-dev 8080:80
    
    ```
    
2.  **Крок 2 (Створення SSH-тунелю на локальному комп'ютері розробника):**
    
    Bash
    
    ```
    ssh -L 8080:127.0.0.1:8080 abilyk_pl@34.118.95.22
    
    ```
