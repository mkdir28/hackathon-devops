# План дій (Action Plan) для команди з 4 людей

Цей план розроблений для швидкого наповнення Канбан-дошки (Jira, Trello, GitHub Projects) стартапу Scout. Він розділений на підготовчу Фазу 0 (розгортання поточного коду "як є") та наступні фази, які передбачають модифікацію коду відповідно до `roadmap.md`.

---

## 👥 Розподіл ролей у команді

* **Dev-1 (AI & Backend Developer):** Розробка серверного коду (Node.js), робота з Redis, Qdrant Vector DB, генерація embeddings.
* **Dev-2 (Frontend & Evals/Security Developer):** Розробка React UI, маскування PII, написання евалюацій та тестів безпеки.
* **DevOps-1 (Cloud & Cluster Engineer):** Інфраструктура K8s (k3d, Kustomize, StatefulSets для Redis/Qdrant, PV/PVC, сховище).
* **DevOps-2 (GitOps & Security Engineer):** GitOps контур (FluxCD, Gateway API / AgentGateway, External Secrets Operator, GHA CI/CD).

---

## 📋 Фази реалізації та Канбан-завдання

### 🏁 ФАЗА 0: Розгортання та CI/CD для поточного коду (Код "як є", без змін)

#### 📝 [TASK-0.1] Локальний кластер k3d та базовий Helm-чарт
* **Виконавець:** DevOps-1
* **Статус:** Виконано (Done)
* **Опис:** Створити локальний Kubernetes кластер k3d, підготувати шаблони локального Umbrella Helm-чарту (`platform/helm/jobmatch`) та розгорнути базові сервіси через FluxCD `HelmRelease`. Створити тестові секрети для LLM API-ключів вручну для дебагу.
* **Критерії приймання:**
  * Кластер k3d запущено.
  * Локальний Helm-чарт успішно створено та валідовано за допомогою `helm lint`.
  * Поди `jobmatch-web` та `jobmatch-api` розгорнуто через HelmRelease.
* **Залежності:** Немає.

#### 📝 [TASK-0.2] Налаштування GitOps (FluxCD) та CI пайплайнів
* **Виконавець:** DevOps-2
* **Статус:** Виконано (Done)
* **Опис:** Налаштувати FluxCD у кластері k3d, що синхронізує ресурси з папки `platform/flux`. Перевірити роботу пайплайну в `.github/workflows/deploy.yml` (збірка, лінт, евалюації у mock-режимі). Зв'язати оновлення образів у `HelmRelease` через `ImageUpdateAutomation`.
* **Критерії приймання:**
  * FluxCD підключено до репозиторію та він успішно синхронізує ресурси.
  * При злитті коду в `dev`/`main` оновлений образ автоматично деплоїться в кластер.
* **Залежності:** TASK-0.1.

#### 📝 [TASK-0.3] Налаштування маршрутизації (Gateway API або Traefik)
* **Виконавець:** DevOps-1 & DevOps-2
* **Статус:** Виконано (Done)
* **Опис:** Налаштувати маршрутизацію через Gateway API або Traefik. Налаштувати `GatewayClass`, `Gateway` та `HTTPRoute` замість застарілого Ingress.
  * `/` -> `jobmatch-web` service.
  * `/api/*` -> `jobmatch-api` service.
* **Критерії приймання:**
  * Застосунок відкривається за єдиною IP-адресою/портом шлюзу.
  * Клієнтська частина успішно відправляє запити на `/api/health` та отримує відповідь від API.
* **Залежності:** TASK-0.2.

### 🛡️ ФАЗА 2: Безпека, Секрети та Контур Evals (Зі зміною коду)

#### 📝 [TASK-2.1] Інтеграція External Secrets Operator (ESO) (Інфра)
* **Виконавець:** DevOps-2
* **Статус:** Виконано (Done)
* **Опис:** Розгорнути ESO в кластері та налаштувати автоматичну синхронізацію API-ключів LLM із GCP Secret Manager (AWS Secrets Manager) у секрети Kubernetes.
* **Критерії приймання:**

#### 📝 [TASK-2.2] Маскування PII та XML-тегування запитів (Код)
* **Виконавець:** Dev-2
* **Статус:** Виконано (Done)
* **Опис:** Написати функцію пре-процесингу резюме, яка видаляє персональні дані перед відправкою до LLM. Огорнути дані користувача в XML-теги для захисту від Prompt Injection.

#### 📝 [TASK-2.4] Налаштування AgentGateway (Envoy) та політик безпеки (Інфра)

* **Критерії приймання:**

#### 📝 [TASK-2.5] Перенаправлення LLM-запитів бекенду через AgentGateway (Код)
