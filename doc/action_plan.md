# План дій (Action Plan) для команди з 4 людей

Цей план розроблений для швидкого наповнення Канбан-дошки (Jira, Trello, GitHub Projects) стартапу Scout. Він розділений на підготовчу Фазу 0 (розгортання поточного коду "як є") та наступні фази, які передбачають модифікацію коду відповідно до `roadmap.md`.

---

## 👥 Розподіл ролей у команді

* **Dev-1 (AI & Backend Developer):** Розробка серверного коду (Node.js), робота з Redis, Qdrant Vector DB, генерація embeddings.
* **Dev-2 (Frontend & Evals/Security Developer):** Розробка React UI, маскування PII, написання евалюацій та тестів безпеки.
* **DevOps-1 (Cloud & Cluster Engineer):** Інфраструктура K8s (k3d, Kustomize, StatefulSets для Redis/Qdrant, PV/PVC, сховище).
* **DevOps-2 (GitOps & Security Engineer):** GitOps контур (FluxCD, Gateway API / AgentGateway, External Secrets Operator, GHA CI/CD).

---

## 📋 Фази реалізації та Канбан-завдання

### 🏁 ФАЗА 0: Розгортання та CI/CD для поточного коду (Код "як є", без змін)

#### 📝 [TASK-0.1] Локальний кластер k3d та базовий Helm-чарт
* **Виконавець:** DevOps-1
* **Опис:** Створити локальний Kubernetes кластер k3d, підготувати шаблони локального Umbrella Helm-чарту (`platform/helm/jobmatch`) та розгорнути базові сервіси через FluxCD `HelmRelease`. Створити тестові секрети для LLM API-ключів вручну для дебагу.
* **Критерії приймання:**
  * Кластер k3d запущено.
  * Локальний Helm-чарт успішно створено та валідовано за допомогою `helm lint`.
  * Поди `jobmatch-web` та `jobmatch-api` розгорнуто через HelmRelease.
* **Залежності:** Немає.

#### 📝 [TASK-0.2] Налаштування GitOps (FluxCD) та CI пайплайнів
* **Виконавець:** DevOps-2
* **Опис:** Налаштувати FluxCD у кластері k3d, що синхронізує ресурси з папки `platform/flux`. Перевірити роботу пайплайну в `.github/workflows/deploy.yml` (збірка, лінт, евалюації у mock-режимі). Зв'язати оновлення образів у `HelmRelease` через `ImageUpdateAutomation`.
* **Критерії приймання:**
  * FluxCD підключено до репозиторію та він успішно синхронізує ресурси.
  * При злитті коду в `dev`/`main` оновлений образ автоматично деплоїться в кластер.
* **Залежності:** TASK-0.1.

#### 📝 [TASK-0.3] Налаштування маршрутизації (Gateway API або Traefik)
* **Виконавець:** DevOps-1 & DevOps-2
* **Опис:** Налаштувати маршрутизацію через Gateway API або Traefik. Налаштувати `GatewayClass`, `Gateway` та `HTTPRoute` замість застарілого Ingress.
  * `/` -> `jobmatch-web` service.
  * `/api/*` -> `jobmatch-api` service.
* **Критерії приймання:**
  * Застосунок відкривається за єдиною IP-адресою/портом шлюзу.
  * Клієнтська частина успішно відправляє запити на `/api/health` та отримує відповідь від API.
* **Залежності:** TASK-0.2.

### 🔄 ФАЗА 1: Впровадження PromptOps та декуплінг скілів/промптів (Зі зміною коду)

#### 📝 [TASK-1.1] Доопрацювання лоадера скілів для читання плоских файлів (Код)
* **Виконавець:** Dev-1
* **Опис:** Модифікувати `app/server/ai/skills/loader.ts` для підтримки читання плоских `.md` файлів з папки `/app/skills` (це необхідно для монтування через плоский Kubernetes ConfigMap).
* **Критерії приймання:**
  * Бекенд успішно зчитує та ініціалізує скіли як з вкладених папок (локально), так і з плоских файлів (у хмарі).
* **Залежності:** TASK-0.3.

#### 📝 [TASK-1.2] Налаштування ConfigMap-монтування скілів у Helm (Інфра)
* **Виконавець:** DevOps-1 & DevOps-2
* **Опис:** Додати в шаблони Helm-чарту генератор `configmap-skills.yaml` за допомогою `.Files.Glob`, налаштувати `volumeMounts`/`volumes` в API-деплойменті та прописати автоматичний Rolling Update через Pod template checksum анотацію.
* **Критерії приймання:**
  * При запуску `scripts/sync-skills.sh` та пуші у Git, Flux автоматично оновлює ConfigMap та викликає миттєвий перезапуск подів API.
* **Залежності:** TASK-1.1.

#### 📝 [TASK-1.3] Додавання Path Filtering у CI-пайплайн
* **Виконавець:** DevOps-2
* **Опис:** Налаштувати кроки в GHA так, щоб при пуші тільки в папку `app/skills/**` кроки перезбирання Docker-образу пропускалися.
* **Критерії приймання:**
  * Зміна промпту оновлює конфіги в K8s без тривалого білду Docker-образу.
* **Залежності:** TASK-1.2.

### 🛡️ ФАЗА 2: Безпека, Секрети та Контур Evals (Зі зміною коду)

#### 📝 [TASK-2.1] Інтеграція External Secrets Operator (ESO) (Інфра)
* **Виконавець:** DevOps-2
* **Опис:** Розгорнути ESO в кластері та налаштувати автоматичну синхронізацію API-ключів LLM із GCP Secret Manager (AWS Secrets Manager) у секрети Kubernetes.
* **Критерії приймання:**
  * Секрет `llm-secrets` заповнюється автоматично з хмарного провайдера.
* **Залежності:** TASK-0.3.

#### 📝 [TASK-2.2] Маскування PII та XML-тегування запитів (Код)
* **Виконавець:** Dev-2
* **Опис:** Написати функцію пре-процесингу резюме, яка видаляє персональні дані перед відправкою до LLM. Огорнути дані користувача в XML-теги для захисту від Prompt Injection.
* **Критерії приймання:**
  * Персональні дані анонімізуються в логах і не шлються в LLM.
* **Залежності:** TASK-1.1.

#### 📝 [TASK-2.3] Розширення Evals-набору та тестів безпеки
* **Виконавець:** Dev-2
* **Опис:** Додати в `evals/dataset.json` тести на злам системи та витік PII. Налаштувати LLM-суддю для автоматичного блокування злиття PR, якщо вразливості виявлено.
* **Критерії приймання:**
  * CI Gate автоматично блокує PR у разі неуспішних тестів безпеки.
* **Залежності:** TASK-2.2.

### 📉 ФАЗА 3: Державні компоненти (Redis Cache & Qdrant Vector DB) (Зі зміною коду)

#### 📝 [TASK-3.1] Підключення Redis та Qdrant як сабчартів (Інфра)
* **Виконавець:** DevOps-1
* **Опис:** Додати `redis` та `qdrant` як залежності в `Chart.yaml` нашого Umbrella Chart, викачати їх сабчарти через `helm dependency build` та налаштувати їх увімкнення та параметри persistence у `HelmRelease` (`values.yaml`).
* **Критерії приймання:**
  * Сабчарти успішно підключено та налаштовано. Бази даних запускаються та працюють у кластері.
* **Залежності:** TASK-0.3.

#### 📝 [TASK-3.2] Інтеграція Redis для кешування пошуків вакансій (Код)
* **Виконавець:** Dev-1
* **Опис:** Інтегрувати Redis-клієнт в API та налаштувати кешування результатів пошуку вакансій на 12 годин для економії токенів.
* **Критерії приймання:**
  * Повторні запити з однаковими параметрами зчитуються безпосередньо з Redis.
* **Залежності:** TASK-3.1.

#### 📝 [TASK-3.3] Впровадження Qdrant для векторного пошуку CV (Код)
* **Виконавець:** Dev-1
* **Опис:** Додати генерацію embeddings для резюме та вакансій. Реалізувати векторний пошук у Qdrant для первинного відбору вакансій за косинусною близькістю.
* **Критерії приймання:**
  * Ембеддінги зберігаються в Qdrant, а пошук повертає ТОП-10 релевантних вакансій для детального LLM-аналізу.
* **Залежності:** TASK-3.1.
